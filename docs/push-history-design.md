# 微信推送记录功能 — 产品设计文档

## 概述

推送记录是「微信推送」配置页面下的一个子功能模块，展示所有通过 iLink Bot 推送到用户微信的消息历史。它让用户清晰了解：什么内容被推送了、推送是否成功、失败的原因是什么。

## 功能定位

| 属性 | 说明 |
|------|------|
| 所属页面 | 系统配置 → 微信推送 (ConfigPanel → PushSection) |
| 前端文件 | `ui-src/src/components/ConfigPanel.jsx` 的 `PushSection()` 函数 |
| 后端接口 | `GET /api/ilink/push-history?type=&status=&limit=50` |
| 数据来源 | 内存通知队列 `_notifications`，筛选 `push_status != "not_pushed"` 的条目 |

## 信息架构

每条推送记录包含以下字段：

| 字段 | 说明 | 展示方式 |
|------|------|----------|
| **type** | 推送类型 | 彩色标签：关键词(橙) / 摘要(蓝) / 公众号摘要(绿) |
| **push_status** | 推送结果 | 绿点+「推送成功」或红点+「推送失败」 |
| **push_time** | 推送时间 | 灰色小字，紧跟状态标签 |
| **title** | 推送标题 | 一行粗体，truncate 超长 |
| **group_name** | 所属群聊 | 灰色副标题行 |
| **content** | 推送正文 | 3行预览（line-clamp-3），可展开 |
| **push_error** | 失败原因 | 仅 `failed` 时显示，红字+分隔线 |

### 类型标签配色体系

```
关键词提醒  →  bg-[#f59e0b]/10 text-[#f59e0b]  (橙，警示性)
群聊摘要    →  bg-blue-500/10 text-blue-600     (蓝，信息性)
公众号摘要  →  bg-brand-green/10 text-green      (绿，资讯性)
```

配色逻辑：**功能语义映射颜色**。关键词是紧急/即时 → 橙；摘要是定时/信息 → 蓝；公众号是订阅/资讯 → 绿。不是随机配色，而是让用户扫一眼就知道这条推送是什么性质。

### 状态指示

```
delivered  →  绿圆点 + "推送成功"    text-brand-green
failed     →  红圆点 + "推送失败"    text-status-error
```

失败时额外展示 `push_error`，比如：
- `"iLink 会话已过期，请重新绑定"` → 用户知道需要重新绑定
- `"消息超过 4000 字符限制"` → 用户知道内容被截断

## 筛选功能

两个下拉筛选器，即时生效（onChange 触发重新请求）：

| 筛选器 | 选项 | 默认值 |
|--------|------|--------|
| 类型 | 全部类型 / 关键词提醒 / 定时摘要 / 公众号摘要 | 全部 |
| 状态 | 全部状态 / 推送成功 / 推送失败 | 全部 |

筛选逻辑：**用户最常见的两个查询意图**——"我最近漏了哪些推送？"（按状态选 failed）和"关键词命中推送了多少次？"（按类型选 keyword_alert）。

右侧有「刷新」按钮，手动拉取最新数据。

## 空状态设计

```
┌─────────────────────────────────┐
│         📦 (Archive icon)       │
│        暂无推送记录              │
│  关键词命中或定时摘要推送后      │
│       会在此显示                │
└─────────────────────────────────┘
```

空状态文案不只是"暂无数据"，而是**告诉用户数据什么时候会产生**，形成功能闭环的认知。

## 列表渲染规则

- **倒序排列**：最新的推送在最前（`reversed(_notifications)`）
- **最大高度 480px**：超出滚动，不撑爆页面
- **hover 效果**：边框从 50% 透明变为 80%，轻量交互反馈
- **content 预览**：`line-clamp-3`，只展示前 3 行，避免长摘要占满屏幕

## 后端数据流

```
关键词命中 / 定时摘要触发
  → add_notification() 记录到 _notifications
  → 尝试 iLink push
  → push 成功: push_status = "delivered", push_time = 当前时间
  → push 失败: push_status = "failed", push_error = 错误信息
  → 不推送:    push_status = "not_pushed"（不会出现在推送记录中）

前端请求 /api/ilink/push-history
  → _handle_push_history() 从 _notifications 筛选
  → 排除 push_status == "not_pushed"（未尝试推送的不算记录）
  → 应用 type / status 筛选
  → 限条数 limit
  → 返回 { ok, records, total }
```

**关键设计决策**：`not_pushed` 的通知不出现在推送记录里。推送记录只记录"尝试了推送"的事件，不管是成功还是失败。没尝试推送的（比如用户选了"不推送"）根本不进入这个视图。

## 设计原则总结

1. **信息密度适中**：每条记录展示 type + status + time + title + group + content预览，6 个维度一卡片，不折叠不隐藏
2. **语义化配色**：橙=紧急、蓝=信息、绿=资讯，颜色即含义
3. **失败可操作**：不只是报错，push_error 告诉用户具体原因和下一步（"请重新绑定"）
4. **筛选即查询**：两个维度覆盖最常见的查询意图，onChange 实时生效
5. **空状态引导**：不只说"没有数据"，还说"数据什么时候会出现"
6. **倒序+限高**：最新优先，但不会撑爆页面布局

## 复用建议

如果要移植到其他项目，核心结构是：

1. 一个带 `push_status` 字段的通知/事件队列
2. 筛选排除未推送条目
3. 前端：类型标签 + 状态指示 + 时间 + 标题 + 内容预览 + 失败原因
4. 两个下拉筛选（type / status）
5. 倒序列表 + max-height 滚动
6. 空状态引导文案
